# 教育知识库前端业务接口文档

> 适用范围：前端先行开发、接口联调前 mock、页面状态设计
>
> 来源：基于 [PLAN.md](/home/ccr/dev/LearningProject/education_brain/docs/PLAN.md) 整理
>
> 说明：本文档优先服务前端开发，不追求 OpenAPI 级别的严谨度。`PLAN.md` 已明确的内容直接沿用；少量字段和返回结构为了便于前端开发做了补齐，这些补齐项在文中标记为“前端协作约定”。

## 1. 文档目标

这份文档给前端解决 4 件事：

1. 知道有哪些接口可用
2. 知道每个接口该怎么传参数
3. 知道前端页面应该拿到什么结构的数据
4. 知道轮询、SSE、错误态、空态该怎么做

当前接口按业务分为 3 组：

- 导入接口：提交导入任务、轮询任务状态
- 查询接口：课程、题目、文档检索
- 问答接口：提交问题、SSE 流式接收答案、读取历史会话

## 2. 基础约定

### 2.1 Base URL

`PLAN.md` 当前只定义了相对路径，未定义统一前缀。

前端协作约定：

- 开发时将接口基地址配置为环境变量：`VITE_API_BASE_URL`
- 文档中所有路径均为相对路径，如 `/search/courses`
- 如果后端后续增加 `/api/v1` 前缀，前端无需改页面逻辑，只改 base URL 即可

### 2.2 Content-Type

- 普通接口：`application/json`
- SSE 接口：`text/event-stream`

### 2.3 鉴权

`PLAN.md` 当前未定义登录鉴权。前端第一版可按“无鉴权”开发。

前端协作约定：

- 暂不要求 token
- 如果后续增加鉴权，优先通过请求头补充，不改业务 body

### 2.4 通用状态枚举

#### 导入任务状态

- `pending`
- `running`
- `partial_success`
- `completed`
- `failed`

#### 文档类型

- `course_doc`
- `project_doc`

#### 问答意图类型

- `course_intro`
- `question_search`
- `doc_search`
- `knowledge_qa`

### 2.5 通用错误响应

前端协作约定：所有非 2xx 响应尽量返回如下结构。

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "file_path does not exist",
    "details": {
      "field": "file_path"
    }
  }
}
```

建议前端统一处理：

- `400`：参数错误，直接提示用户修改输入
- `404`：资源不存在
- `409`：状态冲突
- `500`：服务内部错误
- `503`：依赖服务不可用，如 MongoDB / Milvus / MinIO

## 3. 导入接口

## 3.1 POST /ingest/catalog

### 用途

导入课程目录文件，写入课程系列和课程模块。

### 请求体

```json
{
  "file_path": "data/数据/课程介绍.md"
}
```

前端协作约定：

- 如果前端不提供文件选择能力，也可以直接用默认路径触发导入

### 成功响应

HTTP `202 Accepted`

```json
{
  "task_id": "ingest_catalog_001",
  "task_type": "catalog",
  "status": "pending"
}
```

### 前端处理建议

- 点击“导入课程目录”后立即拿 `task_id`
- 之后每 `1.5s` 轮询一次 `/ingest/tasks/{task_id}`

## 3.2 POST /ingest/questions

### 用途

导入题库文件，解析题库、题目、质量标记。

### 请求体

```json
{
  "file_path": "data/数据/题目资料.md",
  "enable_quality_check": true
}
```

`enable_quality_check` 为前端协作约定字段，用于控制是否展示质量检查入口。后端即使忽略该字段，前端也可以先保留。

### 成功响应

HTTP `202 Accepted`

```json
{
  "task_id": "ingest_questions_001",
  "task_type": "questions",
  "status": "pending"
}
```

## 3.3 POST /ingest/documents

### 用途

导入课程文档或项目文档，执行解析、分块、图片上传、向量化、Milvus 入库。

### 请求体

```json
{
  "file_paths": [
    "data/数据/课程文档/尚硅谷大模型技术之Python1.0.docx",
    "data/数据/课程文档/尚硅谷大模型技术之MySQL1.0.docx"
  ],
  "doc_type": "course_doc",
  "source_mappings": [
    {
      "source_file": "尚硅谷大模型技术之Python1.0.docx",
      "series_code": "general_purpose_programming_foundation",
      "module_code": "general_purpose_programming_foundation_m1"
    }
  ]
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `file_paths` | `string[]` | 是 | 要导入的文件路径列表 |
| `doc_type` | `string` | 是 | `course_doc` 或 `project_doc` |
| `source_mappings` | `object[]` | 否 | 可选的来源映射信息 |

### 成功响应

HTTP `202 Accepted`

```json
{
  "task_id": "ingest_documents_001",
  "task_type": "documents",
  "status": "pending",
  "sub_task_count": 2
}
```

### 前端处理建议

- 文档导入页要支持“批量文件状态”
- 一个主任务下展示多个子任务状态
- 要支持 `partial_success` 展示

## 3.4 GET /ingest/tasks/{task_id}

### 用途

轮询导入任务状态。

### 路径参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `task_id` | `string` | 导入任务 ID |

### 成功响应

```json
{
  "task_id": "ingest_documents_001",
  "task_type": "documents",
  "status": "partial_success",
  "created_at": "2026-04-17T14:00:00+08:00",

    "total": 2,
    "completed": 1,
    "failed": 1
  },
  "sub_tasks": [
    {
      "file": "尚硅谷大模型技术之Python1.0.docx",
      "status": "completed",
      "chunks": 85,
      "warning_count": 1
    },
    {
      "file": "尚硅谷大模型技术之MySQL1.0.docx",

  "progress_logs": [
    {
      "time": "2026-04-17T14:00:01+08:00",

    {
      "time": "2026-04-17T14:01:20+08:00",
```

- 轮询间隔：`1500ms`
- 当状态为 `completed` / `partial_success` / `failed` 时停止轮询

## 4. 查询接口

## 4.1 GET /search/courses

### 用途

查询课程系列、课程详情、模块列表。

### Query 参数

| 参数 | 类型 | 必填 | 说明 |

| `keyword` | `string` | 否 | 关键词，匹配系列名、描述、分类 |
| `audience` | `string` | 否 | 适合人群 |
| `goal_tag` | `string` | 否 | 学习目标标签 |
| `page` | `number` | 否 | 前端协作约定，默认 `1` |
| `page_size` | `number` | 否 | 前端协作约定，默认 `10` |

`GET /search/courses?keyword=Python&page=1&page_size=10`

## 4.2 GET /search/questions

### 用途

按题库、题型、关键词检索题目。

### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `keyword` | `string` | 否 | 题干关键词 |
| `bank_code` | `string` | 否 | 题库编码 |
| `question_type` | `string` | 否 | 题型 |
| `show_quality_flags` | `boolean` | 否 | 是否返回质量标记 |
| `page` | `number` | 否 | 默认 `1` |
| `page_size` | `number` | 否 | 默认 `20` |

### 示例

`GET /search/questions?bank_code=python_bank&question_type=single_choice&page=1&page_size=20`

### 成功响应

```json
{
  "items": [
    {
      "question_code": "python_bank_q001",
      "bank_code": "python_bank",
      "bank_name": "Python 题库",
      "question_type": "single_choice",

      "options": [
        { "label": "A", "content": "func" },
        { "label": "B", "content": "def" },
        { "label": "C", "content": "function" },
        { "label": "D", "content": "lambda" }
      ],
      "answer_key": ["B"],

      "analysis": "Python 使用 def 定义函数。",
      "quality_flags": []
    }
  ],
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total": 1
  }

```

### 前端页面建议

- 题目列表要支持题型筛选
- `quality_flags` 非空时显示风险标签，不要阻止展示

## 4.3 GET /search/documents

### 用途

检索文档片段，返回带来源信息的 chunk。

### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | `string` | 是 | 检索文本 |
| `doc_type` | `string` | 否 | `course_doc` / `project_doc` |
| `top_k` | `number` | 否 | 默认 `5` |

### 示例

`GET /search/documents?query=Python 连接 MySQL&doc_type=course_doc&top_k=5`

### 成功响应

```json
      "chunk_id": "chunk_001",

      "chunk_kind": "code",
      "chunk_text": "使用 pymysql.connect(host, user, password, database) 建立连接。",
      "score": 0.8912,
      "source_mapping": {
        "series_code": "general_purpose_programming_foundation",
        "module_code": "general_purpose_programming_foundation_m2",
        "project_name": null
      },

        {
          "object_key": "documents/doc_python_mysql_001/images/fig1.png",

        }
      ]
    }
  ]

```

### 前端页面建议

- 文档搜索结果页优先展示 `doc_title + section_path + chunk_text`
- `image_refs` 存在时显示缩略图入口

## 5. 问答接口

## 5.1 POST /chat/query

### 用途

统一问答入口。内部会做意图路由：

- 课程查询
- 题目检索
- 文档检索
- 知识问答

### 请求体

```json
{
  "session_id": "session_web_001",
  "question": "Python 怎么连接 MySQL？",
  "mode": "stream",
  "doc_type": "course_doc"
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `session_id` | `string` | 是 | 会话 ID，由前端生成并持久化 |
| `question` | `string` | 是 | 用户输入问题 |
| `mode` | `string` | 否 | 前端协作约定：`stream` 或 `sync`，默认 `stream` |
| `doc_type` | `string` | 否 | 问答时限制文档范围 |

### 流式模式成功响应

HTTP `202 Accepted`

```json
{
  "task_id": "chat_task_001",
  "session_id": "session_web_001",
  "mode": "stream"
}
```

### 非流式模式成功响应

HTTP `200 OK`

```json
{
  "task_id": "chat_task_001",
  "session_id": "session_web_001",
  "intent": "knowledge_qa",
  "answer": "Python 通常通过 pymysql 或 SQLAlchemy 连接 MySQL。",
  "citations": [
    {
      "chunk_id": "chunk_001",
      "doc_title": "尚硅谷大模型技术之Python连接MySQL",
      "section_path": ["第2章 数据库连接", "2.1 pymysql 基础使用"]
    }
  ]
}
```

### 前端处理建议

- 页面默认按 `mode=stream` 处理
- `session_id` 建议保存在 `localStorage`
- 提交成功后立刻连接 `/chat/stream/{task_id}`

## 5.2 GET /chat/stream/{task_id}

### 用途

基于 SSE 的流式答案输出。

### 请求方式

前端使用 `EventSource` 或支持 SSE 的封装建立连接。

### SSE 事件格式

前端协作约定：推荐以下事件类型。

#### `event: status`

```text
event: status
data: {"stage":"retrieving","message":"正在检索相关文档"}
```

#### `event: token`

```text
event: token
data: {"content":"Python "}
```

#### `event: citation`

```text
event: citation
data: {
  "chunk_id": "chunk_001",
  "doc_title": "尚硅谷大模型技术之Python连接MySQL",
  "section_path": ["第2章 数据库连接", "2.1 pymysql 基础使用"]
}
```

#### `event: done`

```text
event: done
data: {"task_id":"chat_task_001"}
```

#### `event: error`

```text
event: error
data: {"message":"LLM 调用失败"}
```

### 前端处理建议

- `token` 事件逐字拼接到回答区
- `citation` 单独维护来源列表
- 收到 `done` 后关闭 SSE 连接
- 收到 `error` 时展示降级提示

## 7. 前端设计风格建议

这部分不是视觉稿，而是给前端一个统一的页面设计方向，避免先做出来的页面后面推倒重来。

### 7.1 整体风格

建议采用“知识工具型后台”风格，而不是营销官网风格。

关键词：

- 稳定
- 清晰
- 信息密度适中
- 强调检索、引用、状态

不建议：

- 大面积装饰性插画
- 过重的渐变和卡片阴影
- 为了“AI 感”做过度炫技动效

### 7.2 页面布局建议

#### 导入页

- 左侧或顶部：导入入口、文件路径/文件列表、文档类型选择
- 中间主区域：任务状态、进度日志、子任务列表
- 重点突出 `pending / running / completed / partial_success / failed`

#### 检索页

- 上方：搜索框 + 筛选区
- 左侧：结果列表
- 右侧：详情区或抽屉
- 文档检索重点展示 `doc_title`、`section_path`、`chunk_text`

#### 问答页

- 左侧可选：历史会话列表
- 中间：消息流
- 底部：输入框
- 回答区要天然支持“流式输出 + 引用块”

### 7.3 组件风格建议

- Tag/Badge：用于 `doc_type`、`question_type`、`quality_flags`、任务状态
- Timeline/Log List：用于导入进度日志
- Split View：用于检索结果和详情联动
- Citation Card：用于问答引用来源
- Empty State：所有列表页都要有空状态，不要只显示空白

### 7.4 文案风格建议

- 以“业务动作”命名按钮，如“开始导入”“重新查询”“查看来源”
- 错误提示优先可理解，不要直接把底层异常原文暴露给用户
- `partial_success` 建议文案为“部分成功”，不要显示成“失败”

## 8. 前后端对接要求

这部分是最关键的。前端即使现在全用 mock，也要按真实接口方式开发。

目标只有一个：

- 后端开发完成后，前端只替换数据来源，不重写页面逻辑

### 8.1 必须分层

前端建议至少拆成三层：

1. 页面层
负责展示和交互，不直接写请求

2. API 层
负责定义每个接口函数，如 `getCourses`、`createDocumentIngestTask`

3. Mock/Real 数据源层
负责决定当前是返回 mock 数据还是真实请求

不建议：

- 在页面组件里直接写 `fetch('/search/courses')`
- 页面直接依赖临时拼出来的 mock JSON 结构

### 8.2 Mock 和真实接口必须共用同一份类型

建议前端先定义好接口出参类型，mock 数据和真实响应都走这套类型。

例如：

```ts
export interface SearchCoursesResponse {
  items: CourseSeriesItem[]
  pagination: {
    page: number
    page_size: number
    total: number
  }
}
```

要求：

- mock 数据返回 `SearchCoursesResponse`
- 真实接口请求回来后，也先映射成 `SearchCoursesResponse`
- 页面只消费 `SearchCoursesResponse`

这样后续即使后端字段名略有变化，也只改 API 层，不改页面层。

### 8.3 不要让页面直接依赖后端的原始字段细节

建议 API 层做一次轻量映射。

例如页面只关心：

- 标题
- 描述
- 标签
- 模块列表

那就不要在页面里到处写：

- `series_title`
- `goal_tags`
- `category_path`

而是可以在 API 层整理成前端更稳定的 ViewModel。

示例：

```ts
export interface CourseCardVM {
  id: string
  title: string
  description: string
  tags: string[]
  moduleCount: number
}
```

### 8.4 SSE 必须提前按真实模式接入

问答页不要先用“假装一次性返回整段文本”的方式开发，否则后面会返工。

建议现在就按真实流程实现：

1. `POST /chat/query`
2. 拿到 `task_id`
3. 建立 `GET /chat/stream/{task_id}` SSE
4. 处理 `status / token / citation / done / error`

即使当前 token 是 mock 生成的，也要按事件流方式喂给页面。

### 8.5 导入页必须按轮询模式开发

导入页不要假设“点击后立刻成功”。

现在就按以下真实模式开发：

1. 提交导入任务
2. 拿到 `task_id`
3. 每 `1500ms` 轮询一次任务状态
4. 根据 `status` 更新 UI
5. 如果是文档导入，展示 `sub_tasks`

### 8.6 所有列表页都要预留空态、错误态、加载态

至少要覆盖：

- `loading`
- `empty`
- `error`
- `partial_success`
- `done`

特别是：

- 文档导入有 `partial_success`
- 文档检索可能“有结果但无图片”
- 问答可能“检索成功但答案生成失败”

### 8.7 分页参数即使后端未实现，也要先保留

当前 `PLAN.md` 里搜索接口是否统一分页还未完全定死，但前端建议先保留：

- `page`
- `page_size`

这样即使第一版 mock 里只有一页，后续接后端也不需要重构查询状态管理。

### 8.8 Mock 切真实接口的切换方式

建议用统一开关，而不是页面内 if-else。

例如：

```ts
const useMock = import.meta.env.VITE_USE_MOCK === 'true'
```

然后在 API 层切换：

```ts
export async function getCourses(params: SearchCoursesParams) {
  return useMock
    ? mockGetCourses(params)
    : http.get<SearchCoursesResponse>('/search/courses', { params })
}
```

要求：

- 页面层不知道当前是 mock 还是真实接口
- 页面层永远只调用统一 API 方法

### 8.9 错误处理要统一

前端不要每个页面各写一套错误处理。

建议统一做两层：

- API 层：把后端错误转成统一错误对象
- 页面层：决定 toast、空态、重试按钮怎么展示

建议统一错误对象结构：

```ts
export interface AppError {
  code: string
  message: string
  retryable?: boolean
}
```

### 8.10 接口命名和文件命名建议

建议 API 层按业务拆文件：

- `api/ingest.ts`
- `api/course.ts`
- `api/question.ts`
- `api/document.ts`
- `api/chat.ts`

mock 数据建议对应拆分：

- `mock/ingest.ts`
- `mock/course.ts`
- `mock/question.ts`
- `mock/document.ts`
- `mock/chat.ts`

这样后续替换真实接口时，不会牵连页面文件。

## 9. 前端 mock 建议

前端未联调前，建议至少准备以下 mock：

### 9.1 导入任务 mock

- `pending`
- `running`
- `completed`
- `partial_success`
- `failed`

尤其要覆盖文档导入的 `sub_tasks`。

### 9.2 查询结果 mock

- 课程：有模块、有相关文档
- 题目：有选项、有答案、有质量标记
- 文档：有 `section_path`、有 `image_refs`

### 9.3 问答 mock

- 流式 token 连续返回
- citation 中途插入
- done 正常结束
- error 异常结束

### 9.4 推荐 mock 原则

- mock 数据字段名必须和接口文档一致
- mock 数据结构必须和真实响应目标结构一致
- 不允许为了页面方便单独造一套“只在前端存在”的临时字段
- 如果页面确实需要额外字段，放到 ViewModel 映射层，不要改接口层结构
