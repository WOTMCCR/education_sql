# 教育知识库真实数据驱动设计方案

> 状态：设计背景文档 / 非当前联调接口文档
>
> 本文档主要解释项目为什么这样建模和分层，适合理解背景，不适合作为当前接口联调依据。

## Summary

基于当前 `data` 目录的真实数据，系统应按“四类数据源、三条查询链路、一个统一导入框架”来设计，而不是把所有内容都当成普通文档做统一向量化。

已确认的数据现状：
- `课程介绍.md`：`219` 个课程系列，`657` 个课程模块，结构高度稳定，适合做强结构化目录数据。
- `题目资料.md`：`73` 个题库，`1752` 道题，结构总体稳定，但存在题型扩展、字段语义漂移和少量脏数据，适合做“结构化题库 + 原文保留”。
- `16` 个课程 `.docx`：正文型教学资料，包含长文本、代码、链接、图片、部分嵌入对象，适合做“文档分块检索”。
- `2` 个项目 `.docx` + `19` 个项目 `.md`：系统设计/实现型资料，含大量图片引用、表格、代码块、流程说明，适合做“项目知识文档检索与问答”。

设计目标按“准生产版”落地：
- 先支持课程介绍、文档检索、题库检索、知识问答四条核心能力。
- 保证来源可追溯、字段可解释、后续可扩展到 PDF/视频/后台管理。
- 不引入过度复杂的多服务编排，但预留任务队列、增量更新、图像增强和后台管理接口。

## Key Changes

### 1. Data Model

统一定义五类核心实体，避免把结构化数据和非结构化文档混在一起：
- `course_series`：来自 `课程介绍.md` 的课程系列，字段包括 `series_code`、`title`、`description`、`category_path`、`audience`、`goal_tags`、`grade_tags`。
- `course_module`：课程系列下的模块，字段包括 `module_code`、`series_code`、`module_title`、`lesson_count`、`study_hours`、`module_desc`、`sort_order`。
- `question_bank`：来自 `题目资料.md` 的题库，字段包括 `bank_code`、`bank_name`、`domain_tags`、`question_count`。
- `question_item`：题目实体，字段包括 `question_code`、`bank_code`、`question_type`、`stem`、`options_json`、`answer_key`、`reference_answer`、`analysis`、`raw_block`、`quality_flags`。
- `knowledge_document` / `knowledge_chunk`：课程文档、项目文档分块结果，字段包括 `doc_id`、`doc_type`、`source_path`、`source_file`、`title`、`domain_tags`、`section_path`、`chunk_text`、`chunk_kind`、`image_refs`、`code_refs`、`table_refs`。

必须增加一张显式映射表，解决“文档与课程目录未自然关联”的真实问题：
- `source_mapping`：将 `source_file/doc_id/bank_code` 映射到 `series_code/module_code/project_name`。
- 默认不要靠文件名硬推断全部归属；先做规则映射，再允许人工补录。

### 2. Ingestion Design

导入必须按源类型分别处理，不走单一解析链路。

`课程介绍.md` 导入：
- 用确定性 Markdown 解析。
- 识别 `## 系列`、系列字段、`### 课程` 下的 3 个模块。
- 直接落库到 `course_series` 和 `course_module`，不做向量化主存，仅为搜索补充检索文本。

`题目资料.md` 导入：
- 用确定性块解析，以 `## 题库` 和 `### question_code` 为边界。
- 每题固定抽取 `题型/题干/选项/答案/解析/raw_block`。
- 对多选答案做标准化，统一拆成数组。
- 对材料分析题、案例分析题、情境分析题保留 `answer_key` 和 `reference_answer` 双字段语义，避免把“作答要求”误当标准答案。
- 对空白选项、重复词、异常标点、字段语义漂移写入 `quality_flags`。

课程 `.docx` 导入：
- 使用 `python-docx` 做主解析，保留段落顺序、样式、表格、图片引用、代码块近似结构。
- 不依赖一次性全文抽取；按段落流式读取并按标题/空行/代码密度做分块。
- 图片和嵌入对象先做元数据登记，不做第一阶段 OCR。
- 课程文档主要用于“知识点检索”和“知识问答引用”。

项目 `.md` / `.docx` 导入：
- Markdown 项目文档按标题层级、表格、代码块、图片引用分块。
- 项目 `.docx` 与课程 `.docx` 同链路，但打上 `doc_type=project_doc`。
- 项目文档需要保留 `section_path`，因为真实查询会落到“某节点流程/某实现步骤”。

### 3. Retrieval and QA Routing

查询层按意图分三路，不能只靠统一 RAG：
- `课程介绍链路`：优先查 `course_series/course_module` 的结构化索引，返回课程列表、适合人群、章节结构、学习目标。
- `题库检索链路`：优先查 `question_item`，支持按题型、题库、领域、关键词过滤，返回题目详情和来源。
- `文档检索链路`：查 `knowledge_chunk`，返回课程/项目/章节/文件来源和片段。

问答路由建议：
- 先做轻量意图分类：`course_intro`、`question_search`、`doc_search`、`knowledge_qa`。
- `knowledge_qa` 再走 Hybrid Retrieval + Rerank。
- 对“有哪些 Python 课程”“查一个选择题”“PyTorch 去噪模块怎么做”这三类问题，分别走不同主链路，而不是全部丢给 LLM。

检索技术路线建议：
- 主数据库：`PostgreSQL`，承载结构化实体、任务状态、映射关系、审计信息。
- 检索引擎：`OpenSearch`，承载全文索引、中文分词、过滤检索、向量字段。
- 向量模型：`BGE-M3`，适配中英混合、代码文本、稠密+稀疏混合检索。
- 重排序模型：`bge-reranker` 或同级中文友好 reranker。
- 问答模型：保持可插拔，首版以中文/代码能力较强的通用模型为主。
- 结论：当前数据规模不需要先上 Milvus；准生产版用 `PostgreSQL + OpenSearch` 足够，且更适合结构化过滤与中文检索。

### 4. Public APIs / Interfaces

建议首版接口按能力边界拆分：
- `POST /ingest/catalog`：导入课程目录。
- `POST /ingest/questions`：导入题库。
- `POST /ingest/documents`：导入课程/项目文档。
- `GET /search/courses`：课程介绍与课程详情查询。
- `GET /search/questions`：题目检索。
- `GET /search/documents`：文档片段检索。
- `POST /chat/query`：统一问答入口，内部做意图路由。
- `GET /ingest/tasks/{task_id}`：导入任务状态。

响应里必须统一返回：
- `content_type`
- `title`
- `source_file`
- `source_path`
- `series_code/module_code/bank_code/question_code` 中的相关字段
- `citations` 或 `matched_chunks`

## Prep Work

上线前必须先做这些前置准备，避免后面返工：
- 建立一份 `data manifest`，登记所有源文件、类型、大小、归属、是否启用。
- 建立 `source_mapping` 初版，把课程文档、项目文档、题库和课程系列做人工可编辑映射。
- 抽样验证 `python-docx` 是否能稳定拿到课程文档的标题层级、表格和图片关系；如果样式丢失严重，再补充 OOXML 级解析。
- 设计 `quality_flags` 枚举，并把题库中的异常答案格式、空白选项、重复词、异常标点归档。
- 确定 OpenSearch 中文分词方案，并用真实问题集做一次课程/题目/文档三类检索验证。
- 准备一套小规模评测集，覆盖课程介绍、题目检索、项目实现问答、课程知识点问答四类场景。
- 明确图片处理策略：首版只保留图片引用与来源；二期再考虑 OCR/图像摘要。
- 明确增量策略：目录和题库按全量覆盖；文档按文件级版本重建索引。

## Test Plan

验收必须基于真实数据，不做空 schema 验收：
- `课程目录解析`：`219` 个系列、`657` 个模块完整入库，字段无丢失。
- `题库解析`：`73` 个题库、`1752` 道题完整入库，所有题均保留 `raw_block`。
- `题型兼容`：至少覆盖单选、多选、判断、简答、编程、阅读理解、材料分析、案例分析、情境分析、计算题。
- `异常兼容`：空白 `C/D` 选项、混合答案分隔符、材料题字段语义漂移都不会导致解析失败。
- `文档导入`：至少用 5 份课程 `.docx`、2 份项目 `.docx`、3 份项目 `.md` 做结构抽样验收。
- `检索效果`：课程介绍类问题命中结构化数据优先；题目类问题直接命中题目实体；项目实现类问题能返回带来源的文档片段。
- `问答引用`：答案必须返回可追溯引用，不能只给自由生成文本。

## Assumptions

- 当前交付目标按“准生产版”设计，不按单机 Demo，也不按多租户平台一次性做满。
- 现阶段真实数据中未发现 PDF 主数据源，因此 PDF/MinerU 不列为首版必要链路，只保留扩展位。
- 图片、嵌入 PPT、复杂图表首版不做多模态理解，只保留引用和来源信息。
- 文档与课程系列之间不存在可靠天然主键，首版接受“规则映射 + 人工补充”的治理方式。
- 首版重点是“导入真实数据并稳定支撑课程介绍、题目检索、文档检索、知识问答”，后台管理、视频检索、权限控制列入后续阶段。
